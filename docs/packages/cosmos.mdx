## Overview

The `@sei-js/cosmos` package provides TypeScript libraries for interaction with Sei using Cosmos standards and interfaces. It offers comprehensive type definitions, encoding utilities, and REST client functionality for all Sei modules.

## Package Structure

The `@sei-js/cosmos` package is organized into three main sub-packages:

### @sei-js/cosmos/types

Provides TypeScript types, enums, and interfaces for all Sei modules.

### @sei-js/cosmos/encoding

Protobuf encoding/decoding utilities, CosmJS Stargate amino converters and registry types.

### @sei-js/cosmos/rest

REST client for querying Sei chain state through the Cosmos REST interface.

## Encoding Package

### Registry and Amino Converters

<CodeGroup>

```typescript Message Registry
import { createRegistry } from '@sei-js/cosmos/encoding';

const registry = createRegistry();

// Register custom message types
registry.register('/sei.tokenfactory.v1beta1.MsgCreateDenom', MsgCreateDenom);
```

```typescript Amino Converters
import { createAminoConverters } from '@sei-js/cosmos/encoding';

const aminoConverters = createAminoConverters();

// Convert protobuf message to amino
const aminoMsg = aminoConverters.toAmino({
  typeUrl: '/cosmos.bank.v1beta1.MsgSend',

});
```

</CodeGroup>

### Protobuf Encoding

<CodeGroup>

```typescript Encode Messages
import { MsgSend } from '@sei-js/cosmos/types/cosmos/bank/v1beta1';

const msgSend: MsgSend = {
  from_address: 'sei1...',
  to_address: 'sei1...',
  amount: [{ denom: 'usei', amount: '100' }]
};

// Encode to protobuf bytes
const encoded = MsgSend.encode(msgSend).finish();

// Decode from protobuf bytes
const decoded = MsgSend.decode(encoded);
```

```typescript Transaction Building

const txBody: TxBody = {

    value: MsgSend.encode(msgSend).finish()

  timeout_height: BigInt(0),
  extension_options: [],
  non_critical_extension_options: []
};

const txRaw: TxRaw = {
  body_bytes: TxBody.encode(txBody).finish(),
  auth_info_bytes: AuthInfo.encode(authInfo).finish(),
  signatures: [signature]
};
```

## Custom Module Integration

### Adding Custom Types

<CodeGroup>

```typescript Custom Message Type
// Define your custom message type
interface MsgCustomAction {
  sender: string;
  action: string;
  parameters: any;
}

// Create protobuf definition
const MsgCustomAction = {
  encode(message: MsgCustomAction, writer = Writer.create()): Writer {
    if (message.sender !== '') {
      writer.uint32(10).string(message.sender);
    }
    if (message.action !== '') {
      writer.uint32(18).string(message.action);
    }
    return writer;
  },

  decode(input: Reader | Uint8Array, length?: number): MsgCustomAction {
    const reader = input instanceof Reader ? input : new Reader(input);
    let end = length === undefined ? reader.len : reader.pos + length;
    const message = { sender: '', action: '', parameters: {} };
    
    while (reader.pos < end) {
      const tag = reader.uint32();
      switch (tag >>> 3) {
        case 1:
          message.sender = reader.string();
          break;
        case 2:
          message.action = reader.string();
          break;
        default:
          reader.skipType(tag & 7);
          break;
      }
    }
    return message;
  }
};
```

</CodeGroup>

## Best Practices

<AccordionGroup>
  <Accordion title="Type Safety">
    - Always use TypeScript types for message construction
    - Validate message fields before encoding
    - Use type guards for runtime type checking
    - Leverage IDE autocomplete for available fields
  </Accordion>

  <Accordion title="Performance">
    - Cache REST client instances
    - Use batch queries when possible
    - Implement proper error handling for network requests
    - Consider using pagination for large result sets
  </Accordion>

  <Accordion title="Error Handling">
    - Handle network timeouts gracefully
    - Validate API responses before processing
    - Implement retry logic for failed requests
    - Log errors with sufficient context
  </Accordion>
</AccordionGroup>
